3. PROBLEM STATEMENT
================================================================================

REAL-WORLD CHALLENGE:
  Employee equity awards (ESOP and RSU) are complex financial instruments that
  require precise tracking, valuation, and tax calculation. Organizations 
  struggle with:
  
  1. MANUAL TRACKING: Excel-based or paper records are error-prone and 
     difficult to scale
  
  2. TAX COMPLEXITY: Calculating taxes on equity exercises requires understanding
     multiple tax regimes, slabs, and calculations
  
  3. MARKET VALUATION: Determining current market prices requires manual lookups
     from multiple sources
  
  4. PROCESS INEFFICIENCY: Exercise requests, approvals, and settlements are
     time-consuming and lack audit trails
  
  5. COMPLIANCE ISSUES: Lack of proper documentation and audit logs

WHO IS AFFECTED:
  - Employees: Struggle to understand their equity holdings and tax implications
  - HR/Admin Teams: Spend significant time managing equity processes manually
  - Compliance Officers: Lack centralized audit trails
  - Finance Teams: Cannot easily track equity-related financials
  - Organizations: Risk non-compliance with equity management regulations

EVIDENCE/EXAMPLES:
  - Companies lose track of unvested options due to manual processes
  - Employees pay incorrect taxes due to miscalculation
  - Approval workflows lack transparency and documentation
  - No centralized location for equity historical data
  - Regulatory audits require extensive manual documentation effort

5. USE CASES
================================================================================

USE CASE 1: Employee Login and Dashboard Access
  ACTORS: Employee
  PRECONDITIONS: Employee has valid credentials
  WORKFLOW:
    1. Employee navigates to login page
    2. Enters email and password
    3. System validates credentials using BCrypt hashing
    4. Generates JWT token on successful authentication
    5. Employee dashboard displays:
       - Overview of all awards
       - Total units vested/unvested
       - Upcoming vesting dates
       - Exercise history
  SUCCESS CRITERIA: Login successful, dashboard shows correct equity data

USE CASE 2: View and Analyze Vesting Schedule
  ACTORS: Employee
  PRECONDITIONS: Employee is logged in, has awards assigned
  WORKFLOW:
    1. Employee navigates to "Awards" section
    2. Selects an award to view details
    3. System displays vesting schedule with:
       - Vesting dates
       - Units vesting at each date
       - Cumulative vested units
       - Status (Pending/Vested)
    4. Employee can see graphical representation
  SUCCESS CRITERIA: Complete vesting schedule displayed accurately

USE CASE 3: Calculate Tax on Exercise
  ACTORS: Employee
  PRECONDITIONS: Employee has vested units, market prices available
  WORKFLOW:
    1. Employee opens Tax Calculator
    2. Selects award type (ESOP/RSU)
    3. Enters market price or system fetches real-time price
    4. Enters number of units to exercise
    5. System calculates:
       - Perquisite value: (Market Price - Exercise Price) × Shares
       - Applicable tax slab
       - Total tax liability
       - Net proceeds
    6. Employee sees detailed breakdown
  SUCCESS CRITERIA: Tax calculated correctly, breakdown shown

USE CASE 4: Submit Exercise Request
  ACTORS: Employee
  PRECONDITIONS: Employee has vested units, completed tax calculation
  WORKFLOW:
    1. Employee clicks "Request Exercise"
    2. System pre-fills market prices and employee data
    3. Employee confirms quantity and details
    4. System calculates exercise costs and taxes
    5. Exercise request submitted with status "Pending"
    6. Admin receives notification
    7. Employee receives confirmation email
  SUCCESS CRITERIA: Request submitted, audit log created, emails sent

USE CASE 5: Admin Review Exercise Request
  ACTORS: Admin/Approver
  PRECONDITIONS: Exercise requests exist in "Pending" status
  WORKFLOW:
    1. Admin views pending exercise requests
    2. Reviews employee details and calculations
    3. Admin can approve or reject request
    4. System updates status and sends email to employee
    5. Audit log recorded with admin action
  SUCCESS CRITERIA: Request approved/rejected, employee notified, audit logged

USE CASE 6: Admin Create Award for Employee
  ACTORS: Admin
  PRECONDITIONS: Employee profile exists
  WORKFLOW:
    1. Admin navigates to award creation
    2. Selects employee
    3. Enters award details:
       - Award type (ESOP/RSU)
       - Grant date
       - Total units
       - Exercise price (for ESOP)
    4. System creates vesting schedule automatically
    5. Audit log records creation
    6. Employee receives notification
  SUCCESS CRITERIA: Award created, vesting schedule generated, notification sent

USE CASE 7: View Audit Log
  ACTORS: Admin
  PRECONDITIONS: Admin is logged in
  WORKFLOW:
    1. Admin navigates to Audit section
    2. System displays all transactions with:
       - Action type (Login, Create, Update)
       - Entity type (Employee, Award, Exercise Request)
       - Employee ID
       - Timestamp
    3. Admin can filter by date or action type
  SUCCESS CRITERIA: Complete audit history displayed

USE CASE 8: Password Reset via OTP
  ACTORS: Employee/Admin
  PRECONDITIONS: User has valid email
  WORKFLOW:
    1. User clicks "Forgot Password"
    2. Enters email address
    3. System generates OTP and sends via email using Brevo API
    4. User receives email with OTP code
    5. User enters OTP and new password
    6. System validates OTP and updates password hash
    7. User receives confirmation email
  SUCCESS CRITERIA: Password reset successfully, email confirmed

9. INSTALLATION & SETUP GUIDE
================================================================================

SYSTEM REQUIREMENTS:

HARDWARE:
  - Processor: Intel Core i5 or AMD equivalent (minimum)
  - RAM: 8 GB (minimum), 16 GB (recommended)
  - Storage: 10 GB free disk space
  - Network: Internet connection for API calls

SOFTWARE - BACKEND:
  - Windows 10/11 or Linux or macOS
  - .NET 8.0 SDK or Runtime
  - SQL Server 2019 or later (or SQL Express)
  - Visual Studio 2022 or Visual Studio Code
  - Git

SOFTWARE - FRONTEND:
  - Node.js 18.x or later
  - npm 9.x or later
  - Angular CLI 17.x
  - Web browser (Chrome, Firefox, Safari, Edge latest versions)

BACKEND SETUP:

Step 1: Install .NET 8.0 SDK
  1. Download from https://dotnet.microsoft.com/download
  2. Run installer and follow prompts
  3. Verify installation: Open PowerShell and run:
     dotnet --version

Step 2: Install SQL Server (or SQL Express)
  1. Download SQL Express from:
     https://www.microsoft.com/en-us/sql-server/sql-server-express
  2. Run installer and follow installation wizard
  3. Select "Developer Edition" workload if installing SQL Server
  4. Note username and password for sa account

Step 3: Clone and Navigate to Backend
  1. Open terminal/PowerShell
  2. Clone repository:
     git clone <repository-url>
  3. Navigate to backend directory:
     cd BackEnd

Step 4: Configure Database Connection
  1. Open appsettings.json in BackEnd folder
  2. Update ConnectionString:
     Locate: "ConnectionStrings": { "DefaultConnection": "..." }
     Update the server name and database name as needed
     Example:
       "Server=LTPHYD052670065\\SQLEXPRESS;
        Database=EquityAwardsDB;
        Trusted_Connection=True;
        TrustServerCertificate=True;"

Step 5: Configure JWT Settings
  1. In appsettings.json, update JWT section:
     "Jwt": {
       "Key": "YOUR_SECURE_SECRET_KEY_HERE",
       "Issuer": "EquiMatrixAPI",
       "Audience": "EquiMatrixUI",
       "ExpiryMinutes": 60
     }

Step 6: Configure Email Settings
  1. Setup Brevo account at https://www.sendinblue.com (now Brevo)
  2. Get API key from Brevo dashboard
  3. Update appsettings.json:
     "EmailSettings": {
       "EmailAddress": "your-email@domain.com",
       "BrevoApiKey": "your-brevo-api-key"
     }
  4. Alternatively, set environment variable:
     $env:BREVO_API_KEY = "your-api-key"

Step 7: Install NuGet Packages
  1. Run command:
     dotnet restore

Step 8: Run Database Migrations
  1. Open Package Manager Console
  2. Run:
     Update-Database
  
  Or using dotnet CLI:
     dotnet ef database update

Step 9: Run Backend Server
  1. Run:
     dotnet run
  
  2. Backend will start on:
     http://localhost:5139 (HTTP)
     https://localhost:7132 (HTTPS)
  
  3. Swagger UI available at:
     https://localhost:7132/swagger

FRONTEND SETUP:

Step 1: Install Node.js
  1. Download from https://nodejs.org (LTS version recommended)
  2. Run installer
  3. Verify installation:
     node --version
     npm --version

Step 2: Install Angular CLI
  1. Run:
     npm install -g @angular/cli

Step 3: Navigate to Frontend Directory
  1. Open terminal and navigate:
     cd FrontEnd

Step 4: Install Frontend Dependencies
  1. Run:
     npm install

Step 5: Configure API Endpoint
  1. Open: src/app/core/services/api.service.ts
  2. Update API base URL:
     baseUrl = 'http://localhost:5139/api'

Step 6: Run Development Server
  1. Run:
     npm start
  
  Or:
     ng serve
  
  2. Frontend will start on:
     http://localhost:4200

Step 7: Access Application
  1. Open browser and navigate to:
     http://localhost:4200
  2. Login page will load

DATABASE SETUP:

Step 1: Verify Database Creation
  1. Open SQL Server Management Studio
  2. Connect to your SQL Server instance
  3. Database "EquityAwardsDB" should be created
  4. Verify tables exist

Step 2: Seed Initial Data (Optional)
  1. Login to application as admin
  2. Navigate to Bootstrap Admin section
  3. Import tax slab data
  4. Create sample employees if needed

Step 3: Backup Database (Recommended)

Issue: Database connection failure

    2. Check connection string in appsettings.json
    3. Verify Windows authentication is enabled
    4. Check firewall settings

Issue: Port already in use
  Solution:

      netstat -ano | findstr :5139  (Find process)
      taskkill /PID <process-id> /F  (Kill process)
    Frontend:
      ng serve --port 4300  (Use different port)

Issue: npm packages not installing
  Solution:
    1. Clear npm cache: npm cache clean --force
    2. Delete node_modules: rm -r node_modules
    3. Delete package-lock.json: rm package-lock.json
    4. Reinstall: npm install

Issue: Angular CLI not found
  Solution:
    npm install -g @angular/cli@latest

Issue: JWT token errors
  Solution:
    1. Verify JWT configuration in appsettings.json
    2. Ensure token hasn't expired
    3. Check Authorization header format: "Bearer <token>"

12. DATABASE SCHEMA & DATA MODELS
================================================================================

DATABASE NAME: EquityAwardsDB
LOCATION: SQL Server (Local or LTPHYD052670065\SQLEXPRESS)

TABLE 1: EMPLOYEES
  Purpose: Store employee information
  Columns:
    - EmpId (INT PRIMARY KEY, IDENTITY(1,1))
    - EmpName (VARCHAR(45), NOT NULL)
    - EmpEmail (VARCHAR(45), NOT NULL, UNIQUE)
    - EmpDOJ (DATETIME, NOT NULL)
    - Password_Hash (VARCHAR(255), NOT NULL)
    - Role (VARCHAR(45), NOT NULL) - Values: "Employee", "Admin"
  
  Relationships:
    - One-to-Many with Awards
    - One-to-Many with ExerciseRequests
    - One-to-Many with VestingSchedules
    - One-to-Many with Notifications
    - One-to-Many with AuditLogs

TABLE 2: AWARDS
  Purpose: Store equity award information
  Columns:
    - AwardId (INT PRIMARY KEY, IDENTITY(1,1))
    - Award_Type (VARCHAR(45), NOT NULL) - Values: "ESOP", "RSU"
    - Grant_Date (DATETIME, NOT NULL)
    - Total_Units (INT, NOT NULL)
    - Exercise_Price (FLOAT, NOT NULL) - Price per unit in USD
    - Employee_EmpId (INT, FOREIGN KEY)
  
  Relationships:
    - Many-to-One with Employees
    - One-to-Many with ExerciseRequests
    - One-to-Many with VestingSchedules

TABLE 3: VESTING_SCHEDULES
  Purpose: Track vesting milestones for each award
  Columns:
    - Vesting_Id (INT PRIMARY KEY, IDENTITY(1,1))
    - Vesting_Date (DATETIME, NOT NULL)
    - Units_Vested (INT, NOT NULL) - Units vesting at this date
    - Cumulative_Vested (INT, NOT NULL) - Total units vested so far
    - Status (VARCHAR(45), NOT NULL) - Values: "Pending", "Vested"
    - Employee_EmpId (INT, FOREIGN KEY)
    - Awards_AwardId (INT, FOREIGN KEY)
  
  Relationships:
    - Many-to-One with Employees
    - Many-to-One with Awards

TABLE 4: EXERCISE_REQUESTS
  Purpose: Track exercise requests submitted by employees
  Columns:
    - RequestId (INT PRIMARY KEY, IDENTITY(1,1))
    - Units_Requested (INT, NOT NULL)
    - Requested_Date (DATETIME, NOT NULL)
    - Status (VARCHAR(45), NOT NULL) - Values: "Pending", "Accepted", "Rejected"
    - CurrentSharePrice (DECIMAL, NOT NULL)
    - ExerciseAmountUsd (DECIMAL, NOT NULL)
    - TaxableGainUsd (DECIMAL, NOT NULL)
    - TaxAmountUsd (DECIMAL, NOT NULL)
    - NetAmountUsd (DECIMAL, NOT NULL)
    - Employee_EmpId (INT, FOREIGN KEY)
    - Awards_AwardId (INT, FOREIGN KEY)
    - VestingScheduleId (INT, FOREIGN KEY, NULLABLE)
  
  Relationships:
    - Many-to-One with Employees
    - Many-to-One with Awards
    - Many-to-One with VestingSchedules

TABLE 5: NOTIFICATIONS
  Purpose: Store notifications sent to employees
  Columns:
    - NotificationId (INT PRIMARY KEY, IDENTITY(1,1))
    - Message (VARCHAR(255), NOT NULL)
    - Type (VARCHAR(45), NOT NULL) - Values: "Info", "Alert", "Warning"
    - Is_Read (BIT, DEFAULT 0)
    - CreatedDate (DATETIME, DEFAULT GETUTCDATE())
    - Employee_EmpId (INT, FOREIGN KEY)
  
  Relationships:
    - Many-to-One with Employees

TABLE 6: AUDIT_LOGS
  Purpose: Track all system actions for compliance
  Columns:
    - AuditLogId (INT PRIMARY KEY, IDENTITY(1,1))
    - Action_Type (VARCHAR(255), NOT NULL) - Values: "Login", "Create", "Update", "Delete"
    - Entity_Type (VARCHAR(255), NOT NULL) - Values: "Employee", "Award", "ExerciseRequest"
    - Entity_Id (INT, NULLABLE) - ID of affected entity
    - Employee_EmpId (INT, FOREIGN KEY)
    - CreatedDate (DATETIME, DEFAULT GETUTCDATE())
  
  Relationships:
    - Many-to-One with Employees

TABLE 7: TAX_COUNTRIES
  Purpose: Store tax country configurations
  Columns:
    - Id (INT PRIMARY KEY, IDENTITY(1,1))
    - CountryCode (VARCHAR(10), NOT NULL) - e.g., "IND", "USA"
    - Description (VARCHAR(500), NULLABLE)
    - ReferenceFxRate (DECIMAL, NOT NULL)
  
  Relationships:
    - One-to-Many with TaxRegimes

TABLE 8: TAX_REGIMES
  Purpose: Store tax regime information

Location: BackEnd/Services/TaxService.cs & ITaxService.cs

Purpose: Calculate taxes on equity exercises and sales

Key Methods:
  - CalculateTax(request): Main tax calculation engine
  - Returns: Total tax, breakdown by slab, effective tax rate

Business Logic:
  1. Retrieves tax slabs from database (ordered by lower bound)
  2. Iterates through slabs progressively
  3. Calculates taxable income in each slab
  4. Applies tax rate to taxable income
  5. Accumulates total tax
  6. Returns detailed breakdown

Tax Calculation Flow:
  Input:
    - Annual Income in USD
    - Award Type (ESOP/RSU)
    - Exercise Price / Market Price
    - Number of Shares
  
  Processing:
    a) For ESOP Exercise:
       - Perquisite Value = (Market Price - Exercise Price) × Shares
       - Total Taxable Income = Annual Income + Perquisite Value
    
    b) For RSU Exercise:
       - Market Price = Exercise Price at Grant (typically 0)
       - Perquisite Value = Market Price × Shares
       - Total Taxable Income = Annual Income + Perquisite Value
    
    c) Tax Slab Application:
       - For each slab (0-10k @ 5%, 10k-50k @ 10%, etc.):
         - Check if income falls in this slab
         - Calculate taxable amount in slab
         - Apply tax rate
         - Add to total tax
  
  Output:
    - Total Tax Amount USD
    - Effective Tax Rate (%)
    - Breakdown by each slab (Lower/Upper bounds, Rate, Taxable, Tax Amount)

TAX CALCULATION ENGINE - DETAILED:

Input Validation:
  - Annual Income: ≥ 0, ≤ 10,000,000
  - Market Price: > 0
  - Exercise Price: ≥ 0
  - Shares: > 0, ≤ 100,000

Calculation Steps:

For ESOP Exercise:
  1. Perquisite Value = (Market Price - Exercise Price) × Shares
  2. Total Income = Annual Income + Perquisite Value
  3. Apply tax slabs progressively:
     a) Query tax slabs ordered by LowerBound ascending
     b) For each slab:
        - Check if Total Income > LowerBound
        - If yes, calculate taxable in this slab:
          Taxable = min(Total Income, UpperBound) - LowerBound
        - Tax in slab = Taxable × Rate
        - Add to cumulative tax
        - If Total Income < UpperBound, break loop

For RSU Exercise:
  1. Exercise Price = 0 (RSU is grant at market price)
  2. Perquisite Value = Market Price × Shares
  3. Rest same as ESOP

For LTCG (Selling - Holding ≥ 12 months):
  1. Capital Gain = (Sale Price - Purchase Price) × Shares
  2. Tax Amount = Capital Gain × 0.125 (12.5% fixed rate)
  3. Net Proceeds = Capital Gain - Tax Amount

For STCG (Selling - Holding < 12 months):
  1. Capital Gain = (Sale Price - Purchase Price) × Shares
  2. Treated as income
  3. Apply tax slabs as in exercise calculation
  4. Net Proceeds = Capital Gain - Tax Amount

   - API rate limit: Use cached data or manual override

   - Invalid symbol: Show error message

NOTIFICATION SYSTEM:

Notification Creation:
  1. Action triggers notification:
     - Award granted
     - Vesting milestone reached
     - Exercise request approved/rejected
     - System alerts
  
  2. Create notification record:
     INSERT INTO Notifications
     (Message, Type, Is_Read, CreatedDate, Employee_EmpId)
     VALUES (message, type, 0, GETUTCDATE(), empId)
  
  3. Fetch unread count:
     SELECT COUNT(*) FROM Notifications
     WHERE Is_Read = 0 AND Employee_EmpId = empId

  4. Mark as read:
     UPDATE Notifications
     SET Is_Read = 1
     WHERE NotificationId = id

Email Notifications:
  1. Queue email request
  2. Format HTML template
  3. Send via Brevo API
  4. Handle failures with retry logic

AUDIT LOGGING:

Entry Creation:
  When action occurs:
  1. Capture action type (Login, Create, Update, Delete)
  2. Capture entity type (Employee, Award, ExerciseRequest)
  3. Capture entity ID (if applicable)
  4. Capture user ID (who performed action)
  5. Capture timestamp (GETUTCDATE() for UTC)
  
  INSERT INTO AuditLogs
  (Action_Type, Entity_Type, Entity_Id, Employee_EmpId, CreatedDate)
  VALUES (actionType, entityType, entityId, userId, GETUTCDATE())

Log Query (Admin View):
  SELECT al.*, e.EmpName
  FROM AuditLogs al
  INNER JOIN Employees e ON al.Employee_EmpId = e.EmpId
  WHERE (CAST(al.CreatedDate AS DATE) BETWEEN @startDate AND @endDate)
    AND (al.Action_Type = @actionType OR @actionType IS NULL)
  ORDER BY al.CreatedDate DESC

CACHING STRATEGY:

Market Prices:
  - Cache duration: 5 minutes
  - Key: symbol + date
  - Invalidation: Time-based (5 min) or manual clear
  - Benefit: Reduce API calls, improve performance

Tax Slabs:
  - Cache duration: Application lifetime
  - Load on startup
  - Invalidation: Manual or on bootstrap update
  - Benefit: Fastest tax calculations

Employee Awards:
  - Cache duration: 1 hour
  - Associated with user session
  - Invalidation: On new award creation
  - Benefit: Reduce database queries

18. LIMITATIONS
================================================================================

CURRENT LIMITATIONS:

1. SINGLE CURRENCY:
   - All calculations in USD
   - No multi-currency support
   - Exchange rate conversion manual only
   - Mitigation: For international use, currency conversion service needed

2. SINGLE TAX REGIME:
   - Primarily India tax regime
   - Other countries tax regimes not built
   - Tax slab configuration limited to one regime at a time
   - Mitigation: Create additional tax regime configurations in database

3. API RATE LIMITING:
   - Alpha Vantage API has rate limits (5 calls/min for free tier)
   - Heavy usage can hit limits
   - No queuing mechanism for excess requests
   - Mitigation: Upgrade to paid API tier or implement request queue

4. MARKET PRICE HISTORY:
   - Limited to 365 days of historical data
   - Cannot calculate holding periods beyond 1 year without external data
   - Mitigation: Implement additional data source for historical data

5. MANUAL VESTING SCHEDULE:
   - Vesting schedules are standardized (4-year cliff/vesting)
   - Cannot configure custom vesting patterns per company
   - Mitigation: Add configuration UI for custom vesting patterns

6. FRONTEND STATE MANAGEMENT:
   - Currently uses RxJS Subjects
   - No centralized state management
   - Scalability issues with large feature set
   - Mitigation: Implement NgRx for state management

7. MOBILE RESPONSIVENESS:
   - UI designed for desktop
   - Mobile experience may not be optimal
   - Tax calculator form may be crowded on small screens
   - Mitigation: Responsive design improvements and mobile-specific components

8. EMAIL DELIVERY FAILURES:
   - No retry mechanism for failed emails
   - Failed emails are not retried
   - Mitigation: Implement queu-based email service with retries

9. DATA EXPORT:
   - Limited export capabilities
   - No PDF generation for reports
   - Export format limited to CSV
   - Mitigation: Implement reporting engine with PDF generation

10. PERFORMANCE OPTIMIZATION:
    - No pagination on large lists
    - All records loaded at once
    - Potential memory issues with thousands of records
    - Mitigation: Implement server-side pagination

11. SECURITY HARDENING:
    - Rate limiting not implemented on API endpoints
    - No protection against brute force attacks
    - JWT secret key in appsettings.json (not recommended for production)
    - Mitigation: Implement secrets management and rate limiting

12. TESTING COVERAGE:
    - No automated unit tests included
    - No integration tests
    - Manual testing only
    - Mitigation: Implement comprehensive test suites

13. BATCH OPERATIONS:
    - Cannot perform bulk exercises
    - Cannot batch create awards
    - One-at-a-time operations only
    - Mitigation: Add bulk operation endpoints

14. VESTING SCHEDULE FLEXIBILITY:
    - Cannot modify vesting schedule after creation
    - Cannot handle acceleration/special vesting events
    - Mitigation: Add vesting modification endpoints

15. NOTIFICATION SCHEDULING:
    - Cannot schedule notifications for future dates
    - All notifications immediate
    - Mitigation: Implement scheduler service

16. AUDIT LOG RETENTION:
    - No automatic pruning of old logs
    - Will grow indefinitely
    - Mitigation: Implement log archival and retention policies

17. CONCURRENT REQUEST HANDLING:
    - No handling of simultaneous exercise requests
    - No locking mechanism for double-spending
    - Mitigation: Implement optimistic/pessimistic concurrency control

18. TIME ZONE SUPPORT:
    - All dates in UTC
    - User local time zone not considered
    - Mitigation: Add time zone configuration per user

19. DOCUMENTATION:
    - API documentation is Swagger auto-generated
    - No comprehensive API documentation guide
    - Frontend component documentation limited
    - Mitigation: Create detailed API documentation

20. ACCESSIBILITY:
    - Limited WCAG compliance
    - Color contrast may not meet standards
    - No screen reader optimization
    - Mitigation: Implement accessibility improvements (WCAG 2.1 AA)

KNOWN ISSUES:

1. Token Expiry:
   - User is logged out after 60 minutes
   - No automatic token refresh
   - Solution: Implement token refresh mechanism

2. Password Reset Email:
   - OTP validity is 10 minutes
   - If user misses email, must request again
   - Solution: Add resend OTP button with rate limiting

3. Exercise Request Calculation:
   - Market price is point-in-time
   - If user waits, market price changes not reflected
   - Solution: Show price refresh time, allow user to refresh

4. Vesting Schedule Generation:
   - Cannot handle non-standard vesting patterns
   - Cliff vesting not flexible
   - Solution: Add advanced vesting configuration

5. Tax Calculation Rounding:
   - Rounding may cause minor discrepancies (< 1 USD)
   - Due to successive tax slab calculations
   - Solution: Implement banker's rounding (round-half-to-even)

MOCK DATA USED:

- Tax Slabs: Pre-configured in database
- Employee Grades: Fixed values (Employee, Admin)
- Award Types: ESOP, RSU only
- Notification Types: Info, Alert, Warning only
- Exercise Status: Pending, Accepted, Rejected only

- ~1,000 employees

Beyond these numbers:

Recommendations for scaling:

- Add caching layer (Redis)
- Implement API pagination
- Use NgRx for state management
- Consider microservices architecture

APPENDIX A: GLOSSARY
================================================================================

ESOP: Equity Stock Option Plan - Right to purchase shares at fixed price
RSU: Restricted Stock Units - Conditional ownership that vests over time
Vesting: Process of earning the right to shares over a period
Exercise: Purchasing shares by using an option at exercise price
Perquisite: Taxable benefit from exercising options
LTCG: Long-Term Capital Gains - Tax on sale of assets held >12 months
STCG: Short-Term Capital Gains - Tax on sale of assets held <12 months
JWT: JSON Web Token - Secure authentication token
API: Application Programming Interface - Communication protocol
OTP: One-Time Password - Temporary security code
RBAC: Role-Based Access Control - User permission system
BCrypt: Password hashing algorithm
Alpha Vantage: Stock market data API
Brevo: Email service provider (formerly Sendinblue)
